# 📖 StoryFactory (小说制造工厂)

> **定位**：一个基于 **Spring AI Alibaba** 的自动化、模块化长篇小说创作流水线系统。
> **口号**：从灵感到完本，用工程化思维解决 AI 写长篇小说的“逻辑崩坏”与“失忆”痛点。

## 🎯 一、 解决的核心痛点与工程化解法 (Why & How)

目前市面的 AI 写作工具大多停留在“单次对话”的玩具阶段，无法满足**网文作者/工作室矩阵化量产**的需求。本项目通过以下工程化手段解决两大核心痛点：

### 痛点一：长篇连载的“逻辑一致性” (Consistency) 崩坏
*   **现象**：AI 写短篇很强，写长篇（20万字以上）必定出现上下文遗忘、人物死亡后复活、物品丢失等问题。
*   **根本原因**：现有项目大多只是简单拼接前文上下文，极易超出 Token 限制且费用高昂。
*   **StoryFactory 的解法：世界状态机 (World State Machine) + RAG 记忆库**
    1.  **实体状态机**：在数据库中维护动态的 `character_state`（如：主角当前境界、持有物品、所处位置）。
    2.  **记忆压缩入库**：每生成完一章，利用 Spring AI Hook 触发“总结 Agent”，抽取关键事件（NER）并转化为向量，存入 Milvus/Chroma。
    3.  **精准召回**：生成新章节前，根据大纲意图，系统先去数据库查询主角当前状态，并去向量库检索相关的历史伏笔，动态拼接入 Prompt。

### 痛点二：风格同质化，缺乏“爽点” (Style & Pacing)
*   **现象**：AI 生成的文本像“说明文”，平铺直叙，缺乏网文特有的“黄金三章”、“装逼打脸”、“情绪拉扯”节奏。
*   **根本原因**：大模型倾向于“端水大师”式的平衡输出，缺乏网文结构化数据的微调与强引导。
*   **StoryFactory 的解法：剧本模板引擎 (Script Template) + 多 Agent 拆步生成**
    1.  **强约束模板**：系统内置多种网文套路模板（如：退婚流、神豪流）。例如“打脸模板”被严格规定为：20%反派嘲讽 -> 20%路人压抑 -> 40%主角亮底牌震惊全场 -> 20%冷漠收场。
    2.  **Graph 多节点编排**：利用 Spring AI Alibaba Graph，不让 AI 一次性写完一章，而是按照模板的步骤，像拼乐高一样拆分成多个节点分布生成，确保每一步的情绪拉扯都到位。
    3.  **爽点审查**：引入 Review Agent 评估情绪反差，不达标则触发状态机回滚重写。

## 🚀 二、 核心功能与架构设计 (基于 Spring AI Alibaba)

本项目深度依赖 **Spring AI Alibaba** 的企业级能力，构建了以下核心模块：

### 1. 基于 Graph 的多智能体流水线 (Agentic Workflow)
借助 Spring AI Alibaba Graph 框架，将小说创作抽象为多 Agent 协作的有向无环图 (DAG)：
*   **Outline Agent (大纲智能体)**：将主线剧情拆解为带有强约束的 `情节节点(Plot Points)`。
*   **Draft Agent (草稿智能体)**：严格按照情节节点生成 3000 字正文。
*   **Review Agent (审核智能体)**：校验字数、敏感词以及逻辑连贯性，不合格则触发 Graph 的回滚重写逻辑。

### 2. 动态记忆与增强检索系统 (Memory & RAG)
*   **记忆压缩 Hook**：利用 Spring AI Alibaba 的模型 Hook 机制，在每章生成后自动提取摘要、新人物、关键物品。
*   **向量化存储**：使用 Spring AI Alibaba 丰富的 VectorStore 支持（Milvus / Elasticsearch），构建小说专属的“世界观知识库”。
*   **精准召回增强**：在生成新章节前，利用 RAG 机制检索历史伏笔，动态拼接入 Prompt，彻底解决 AI 失忆。

### 3. 高并发异步调度中心 (Async Task Scheduler)
*   摆脱单线程等待，支持多本书籍、多章节的后台排队异步生成。
*   提供进度监控和任务状态机管理（PENDING, GENERATING, DONE, FAILED）。

## 🏗️ 三、 技术栈选型 (Tech Stack)
*   **核心框架**：Spring Boot 3.x, JDK 17
*   **AI 编排引擎**：**Spring AI Alibaba (含 Graph 模块)**
*   **大模型底座**：DeepSeek (主推), 阿里云通义千问 (Qwen)
*   **数据与检索**：MySQL (业务数据) + Milvus/Chroma (向量数据库)
*   **前端交互**：Vue 3 + Element Plus (提供基础的剧本人机交互工作台)

## 🗺️ 四、 演进路线图 (Roadmap)

*   [x] **Phase 0 (PoC)**: 验证基础 RAG 记忆注入解决“AI失忆”的可行性。
*   [ ] **Phase 1 (MVP)**: 引入 Spring AI Alibaba Graph，完成单本小说“大纲->正文->记忆抽取”的完整 Agent 流水线。
*   [ ] **Phase 2 (人机协作)**: 开发前端控制台，支持作者手动修改大纲/剧情节点，AI 根据修改自动往下续写。
*   [ ] **Phase 3 (矩阵化量产)**: 完善异步队列，支持多账号、多本书籍并行生成与平台自动分发。
